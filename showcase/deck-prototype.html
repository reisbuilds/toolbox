Context

The Problem

AI can write a design brief — it just can't hold five cognitively incompatible modes in the same context window without losing coherence.
  • Discovery, research, specification, verification, and synthesis each require a different epistemic posture — one reading, one searching, one building, one auditing, one assembling
  • A single long prompt collapses all five into one completion with no structured handoff and no way to correct drift between phases
  • The result reads well in isolation and breaks down across sections — contrast ratios that don't match the stated visual language, decisions that ignore the problem named in paragraph one
2 / 11
Solution

What Was Built

toolbox — two multi-agent orchestration pipelines that decompose AI generation tasks into discrete phases, enforcing structured handoffs instead of letting context drift.

  • /design-lifecycle — classified codebase genre, real-world UI references, hex token values with rationale, component hierarchy, and a six-check verification pass before synthesis
  • /project-showcase — comprehensive documentation, a case study with concrete decisions, and a Marp slide deck — all validated before the deck is written
  • Each pipeline extracts named structured blocks from every agent and re-injects them downstream as explicit, labeled data contracts
  • Both run as Claude Code skills — one slash command in any project directory

Architecture

How It Works

Layer Component Role
Orchestration SKILL.md per skill Sequences agents, routes named output blocks, holds zero domain knowledge
Discovery Thinking Partner Reads codebase signals, classifies genre and maturity, produces the central data contract
Execution Maker / Inspiration / Verifier Domain-specific agents that consume the contract and produce artifacts
Quality Gate Verifier / Validator Runs discrete checks before synthesis — failures are named, not inherited
Synthesis Amplifier Assembles final output with explicit anti-AI-voice tone rules

Decision 01 / 03

The Zero-Knowledge Orchestrator

Question Should the orchestrator understand design or documentation, or just coordinate agents that do?
Choice SKILL.md carries no domain knowledge — it sequences and routes named structured blocks, nothing else. All domain knowledge lives in agent files.
Why it matters Coordination logic stays stable regardless of what any agent does — updating the Verifier's WCAG criteria requires editing exactly one file, not touching the orchestration layer.
5 / 11
Decision 02 / 03

Pre-Reading All Agent Files at Startup

Question When should the orchestrator learn what each downstream phase needs?
Choice Read every agent file in a single pass before spawning any agent, giving the orchestrator the full pipeline in scope before execution begins.
Why it matters The orchestrator knows Phase 5 needs both Phase 1 and Phase 2 output before Phase 1 has run — making output routing correct by construction, not discovered ad hoc at each handoff.
6 / 11
Decision 03 / 03

Named Structured Output Contracts

Question How do you pass information between agents without letting context drift or balloon?
Choice Each agent produces a named structured block with required fields. The orchestrator extracts and re-injects these blocks verbatim, labeled, into downstream agents.
Why it matters A 200-token structured block instead of a 3,000-token history keeps downstream agents focused. The Verifier audits design against the original Project Brief — not just the Maker's output in isolation.

Challenges

What Was Hard

Contract-first sequencing — the mistake that cascaded
Individual agent prompt quality was iterated before interface contracts were locked. Every output format change cascaded into rewrites of three downstream prompts. Define and lock the data contracts first.
Preventing cognitive mode bleeding between agents
Each agent receives exactly what it needs, not the full conversation history. Discovery agents don't generate solutions. Verification agents don't author content. Mode boundaries enforced structurally.
Making output human-readable, not AI-readable
Amplifier agents carry explicit prohibited-pattern rules: no process meta-commentary, no adjectives where values belong. Named prohibited patterns work; vague tone guidance does not.

Outcomes

Results

  • Project ran both skills on its own codebase — DESIGN_BRIEF.md generated by /design-lifecycle, this deck generated by /project-showcase
  • Validation Report passed all five checks with zero critical findings: accuracy, voice and tone, completeness, specificity, and deck readiness
  • Zero-buzzword audit: no instances of "leveraged," "robust," "seamless," or "best practices" in either artifact
  • DESIGN_BRIEF.md names specific CSS variable values, specific contrast ratios, and specific pixel dimensions — anti-AI-voice rules working
  • Six-check graduated Verifier structure (pass / warn / fail per check) gives actionable prioritization, not a binary result

Forward

What's Next

  • Add schema validation on named output blocks so format drift fails at extraction time — a lightweight JSON wrapper around required fields catches this without changing the natural-language interface
  • Build a retry path for warn-level and fail-level Verifier findings before the Amplifier runs, rather than surfacing open items in DESIGN_BRIEF.md for the developer to resolve manually
  • Add a codebase-aware entry point for projects with existing design systems, where the Maker extends and audits current token conventions rather than generating a parallel system
