The pipeline

Each phase owns one cognitive mode

Cognitive mode bleeding is the failure mode. When you ask one prompt to discover, specify, and verify simultaneously, something always gets dropped. Usually the unglamorous stuff — error states, degraded states, the contrast ratios that fail at 4.2:1.

The problem

A single prompt cannot hold discovery, research, specification, verification, and synthesis at once. Something always collapses.

Single-prompt design generation produces output that sounds authoritative. It is structurally broken. The model generates tokens and components simultaneously, so it cannot inspect its own token definitions when writing component suggestions.

What you get when context collapses:

Tokens that don't reference the inspiration sourced two phases earlier
Components that don't map to the core flows that were identified
Contrast ratios cited in prose but never calculated against actual hex values
Empty states described and then missing from every screen layout

How it works

Three decisions that define the architecture

Decision 01

The orchestrator reads everything first

SKILL.md reads all five agent files before spawning any sub-agent. The full pipeline is in context before execution begins — not discovered phase by phase. This enables cross-phase reasoning, not just sequential execution.

Decision 02

The Project Brief travels to every agent

Not just the next agent. The Verifier and Amplifier both receive the original upstream classification — genre, platform, maturity, core flows. The Verifier audits against the problem definition, not against its own reading of the output.

Decision 03

The Verifier is mandatory, not optional

Most pipelines: generate → deliver. This one: generate → verify → deliver. Failures caught before the Amplifier can be fixed. Failures that reach DESIGN_BRIEF.md become the implementation standard. The check has to happen while it can still block the output.

Key insight

The orchestrator contains zero design knowledge

SKILL.md sequences and routes. Nothing else. All domain knowledge lives in the agent files. The consequence: individual agents are replaceable without touching the orchestrator. The coordination layer is stable precisely because it is empty of the thing it coordinates.

Data contracts

Agents need typed inputs, not transcripts

Each agent receives exactly what its task requires — a named, structured block extracted from upstream output — not the full conversation history. A 200-token Project Brief is not the same as a 3,000-token conversation thread. The latter buries the signal.

Phase 04

Six checks. Before anything ships.

AI can generate structure. The Verifier checks coherence — before the Amplifier makes it institutional memory.

Maker → Verifier → Amplifier    not: Maker → Amplifier

✓
UI state coverage Empty, loading, error, success, and partial data states — per core flow
✓
WCAG AA contrast 4.5:1 normal text · 3:1 large text and UI components
✓
Touch targets 44×44pt iOS · 48×48dp Android — buttons and icon-only elements
✓
Token consistency Component references cross-checked against defined token names
✓
Component coverage Every core flow in the Project Brief has a screen layout and feature components
✓
Inspiration alignment Color direction, typography approach, and density match the Inspiration Summary

Output

Four files. Committed to disk. Versioned.

Not ephemeral chat responses. Files you can check into the repo, review in a PR, and reference for months. The project's .design/ directory becomes a versioned design system baseline.

.design/design-tokens.md
Color palette with hex values and rationale · type scale · spacing system · border radius · elevation — every value grounded in the genre and Inspiration Summary
Maker
.design/component-suggestions.md
Component hierarchy (primitives → composed → feature) · ASCII wireframes per core flow · interaction patterns for all five states: empty, loading, error, success, partial data

Verifier
DESIGN_BRIEF.md
Canonical handoff artifact written for human designers and developers — design philosophy, full token spec, component spec, verification status, decisions & rationale, next steps
