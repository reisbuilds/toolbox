# toolbox — Case Study

**Role:** Solo
**Status:** MVP

---

## TL;DR

- Single-prompt AI generation produces output that reads well locally but breaks down across phases — tokens that don't satisfy the contrast ratios mentioned two paragraphs later, case study decisions that contradict the stated problem, slides with no connection to the narrative — because no single completion can hold five cognitively incompatible modes simultaneously.
- I built two multi-agent orchestration pipelines that decompose those tasks into discrete phases, extract named structured output blocks from each agent, and re-inject them downstream as explicit data contracts instead of context that drifts.
- The key architectural decision was separating the orchestrator from all domain knowledge: `SKILL.md` in each skill is a pure coordination layer — it sequences and routes, carries nothing else — which makes agents individually replaceable without touching the sequencing logic.
- The output validates itself: the project has run both skills on its own codebase, the Validation Report passed all five checks with zero warnings, and the `DESIGN_BRIEF.md` was generated by the same pipeline it describes.

---

## Context — The Situation

Most developers who need design direction or portfolio artifacts do one of three things: hire a specialist, copy something that looks close enough, or ask an AI to generate the full output in one shot. The first is expensive and slow to spin up. The second produces derivative work with no reasoning behind it — nothing to hand to a contractor or revisit when the UI needs to grow. The third produces output that sounds authoritative until you try to build from it: contrast ratios that don't clear WCAG AA, component names that reference tokens never defined, a case study narrative that describes a problem in paragraph one and ignores it for the next four sections.

The developer this project is for has a working codebase and no designer, no technical writer, and no clear starting point. They are not looking for aesthetic inspiration — they need a token system they can implement, a component list mapped to their actual user flows, and a design brief they can hand to a contractor. Or they need documentation, a case study, and a slide deck to present a project they built. Most AI tools respond to these needs with one very long prompt, get confident-sounding output, and stop there. toolbox is what happens when you treat both of those tasks as pipelines instead.

---

## Goals

**Functional goals:**

- Classify the application from codebase signals before generating any design recommendations
- Source real-world UI references from actual products in the same genre, not generic design advice
- Generate concrete token values — hex codes, type scale numbers, spacing increments — with rationale sentences per decision
- Verify the design output against six discrete checks before synthesis
- Produce portfolio documentation, a case study, and a slide deck from any codebase
- Validate both showcase artifacts through a five-check quality gate before the deck is written

**Quality goals:**

- Each agent receives exactly what it needs — not the full conversation history
- Quality gates run before synthesis, not after
- The orchestration layer holds zero domain knowledge
- Output format per agent is an interface contract, not documentation
- Final artifacts are written for humans to act on, not for the pipeline to consume

---

## The Problem

The failure mode I was solving is not a model capability problem. It is a task architecture problem. When you write "generate a complete design system for this app" or "write a portfolio case study for this project" in a single prompt, you are asking the model to inhabit five cognitively incompatible modes at once: understand the codebase, research analogous products, specify artifacts, verify consistency, and synthesize a coherent document. That is a pipeline. Collapsing it into a single completion is why the output breaks.

What cross-phase incoherence looks like in practice: the Inspiration phase identifies a cool-toned, restrained visual language from production apps in the same genre, but the token generation — happening in the same context window — produces a warm, high-saturation palette because the generation momentum moved that direction. The case study identifies "reducing developer friction" as the core problem in paragraph one, then discusses architecture decisions for the next three sections without mentioning developer friction once. The slide deck's "The Problem" slide says something different than the case study's "The Problem" section. Each piece sounds right. They don't fit together.

The deeper issue is that each phase requires a different epistemic posture. Discovery means reading and classifying — deliberately not generating solutions yet. Research means searching and synthesizing external references — not producing specifications. Specification means building on a fixed classification and fixed references — not re-deriving them. Verification means checking output against an external standard — not authoring more content. Synthesis means assembling everything into a coherent document — not second-guessing earlier phases. Forcing one context window to do all five simultaneously is not a matter of prompt length or model quality. The architecture itself produces the incoherence.

## Process & Decisions

### The Zero-Knowledge Orchestrator

I could have written one very long prompt. I chose not to, because prompt length is not the same as task focus. A 5,000-word prompt that asks for all five phases in sequence is still one completion — every section generates with the full original context active and no structured handoff between phases.

A pipeline orchestrator changes the information flow. `SKILL.md` in each skill is a pure coordination layer: it reads all agent files in a pre-pass, then sequences agents serially — extracting named structured blocks from each output and re-injecting them into downstream agents. It does not generate design content. It does not evaluate design quality. It sequences and routes. All domain knowledge lives in the agent files.

The cost is real: five sequential sub-agent calls take longer than one completion, and the startup pass reads all agent files before anything executes. But there is a boundary here that matters. The pipeline does not just produce more structured output — it produces output where each phase uses the previous phase's result as a grounded input rather than as context it might drift away from in the same completion. That distinction is worth the latency.

The other benefit of this separation: agents are individually replaceable. Updating the Verifier's WCAG criteria, changing the Amplifier's tone rules, adding a new check to the Validator — each requires editing one agent file. The orchestrator's behavior does not change. Coordination logic is stable because it is empty of domain concerns.

### Pre-Reading Agent Files at Startup

The orchestrator reads every agent file in a single pass before spawning the first sub-agent. This gives it the full pipeline in scope before execution begins.

The alternative would be to read each agent file immediately before spawning that agent. That approach works, but it means the orchestrator discovers each downstream phase's requirements only when it arrives there — which makes cross-phase routing fragile. By reading all files upfront, the orchestrator can reason about the full pipeline: it knows that Phase 5 needs both the Phase 1 Project Brief and the Phase 2 Inspiration Summary before Phase 1 has even run. This enables correct output routing without ad-hoc inspection at each handoff.

The tradeoff is a longer startup pass before any visible progress. That is an acceptable cost for a skill that runs infrequently on deliberate demand rather than in a hot execution loop.

### Named Structured Output Contracts

The Project Brief is the central data contract in `/design-lifecycle`. The Thinking Partner produces it in Phase 1 with required named fields: genre, platform, maturity, core user flows, tone keywords, design problem statement. The orchestrator extracts that block verbatim and re-injects it — labeled, unchanged — into Phases 3, 4, and 5. The Project Profile plays the same role in `/project-showcase`.

This matters for a specific reason. The Verifier audits design output against the Project Brief — not against the Maker's recommendations in isolation. Without the Project Brief flowing downstream, the Verifier checks whether the design is internally self-consistent. With it, the Verifier checks whether the design is consistent with what the application actually is. Those are different audits with different failure modes.

The fragility is also real: extraction depends on agents producing exact heading names and block structure. If the Thinking Partner drifts from its required format, downstream agents receive an incomplete contract and the quality signal breaks. Every agent's output format is specified as an interface contract — not documentation — and that is the only enforcement mechanism available in a natural-language orchestration system. There is no schema validation. Format drift fails silently.

I chose a 200-token structured block over passing a 3,000-token conversation history because focused context produces more focused output. A downstream agent that receives exactly the fields it needs — classified genre, identified flows, tone keywords — does not need to re-derive or re-interpret those findings. It acts on them. That compactness is a deliberate tradeoff: it means losing the reasoning that produced the contract, but it keeps each agent operating in its own domain rather than re-processing upstream reasoning.

### Contract-First Sequencing — The Mistake I Made

The data contract definition was the hardest problem in the project, and I solved it too late. I started by iterating on individual agent prompt quality: writing the Thinking Partner prompt, running it, improving the output, then writing the Inspiration agent, and so on. The prompts improved — but each time I iterated on the Thinking Partner's output format, I had to update the downstream prompts that depended on it. When I decided the Project Brief needed a separate "Design Problem Statement" field, I rewrote parts of three agent prompts to account for the change.

The correct sequencing was to define the data contracts first: exactly what fields the Project Brief contains, exactly what structure the Inspiration Summary follows, exactly what the orchestrator extracts and re-injects at each handoff. With the contracts locked, writing each agent's output format section becomes a matter of matching a pre-defined schema rather than negotiating it during iteration. I worked from the inside out — improving individual agents — when I should have worked from the outside in, starting with the interfaces between them.

### Anti-AI-Voice Tone Rules

Both Amplifier agents carry explicit tone rules: no meta-commentary about the process, specific values not adjectives, no hedging language. These rules exist because language models degrade handoff artifacts in predictable and correctable ways.

The predictable failure modes: describing the generation process ("The AI analyzed the codebase and determined..."), using adjectives where values should appear ("comfortable padding" instead of "16px padding"), and qualifying every recommendation with uncertainty hedges ("might consider", "could potentially"). These are not random errors — they are recurring patterns that appear in AI-generated handoff artifacts across domains.

The tone rules name and correct each failure mode directly in the prompt. This approach came from observing that vague instructions like "write in a professional tone" do not prevent these patterns; they require specific named rules. The Amplifier prompts now read less like tone guidance and more like a list of prohibited patterns — which is precisely what makes them work.

## The Solution

You run `/design-lifecycle` or `/project-showcase` in any project directory. The orchestrator reads all agent files in one pass before spawning anything. Then it sequences the pipeline, extracting named structured blocks from each agent's output and injecting them into the next agent as explicit, labeled context.

For `/design-lifecycle`: the Thinking Partner classifies the codebase and produces the Project Brief. The Inspiration agent sources real product UI references in the same genre and produces the Inspiration Summary. The Maker generates `design-tokens.md` — hex values with rationale sentences, type scale, spacing system, border radius, elevation — and `component-suggestions.md`, a component hierarchy mapped to the user flows identified in Phase 1. The Verifier runs six discrete checks and produces a pass/warn/fail report for each. The Amplifier synthesizes everything into `DESIGN_BRIEF.md`, written for humans to implement from, with explicit tone rules preventing AI-voice degradation.

For `/project-showcase`: the Thinking Partner reads the codebase deeply — tech stack, architecture, interesting decisions, inferred challenges, code quality signals — and produces the Project Profile. The Maker writes `DOCUMENTATION.md` and `CASE_STUDY.md`. The Validator runs five checks — accuracy against the Profile, voice and tone, completeness, specificity, and narrative arc for the deck — and writes a Validation Report. The Amplifier writes a Marp slide deck renderable to PDF or HTML with a single CLI command.

Three headline moments:

1. **The orchestrator reads all agent files before spawning any agent.** Every phase has the full pipeline in scope before execution begins. The orchestrator routes outputs correctly because it knows what each downstream agent needs before any of them run.

2. **The Verifier and Validator run before synthesis.** A fail-level finding means the Amplifier either surfaces it explicitly as an open item or — in future iterations — stops the pipeline. Failures caught before synthesis can be named and addressed. Failures inherited after synthesis become institutional memory.

3. **The project has run both skills on itself.** `DESIGN_BRIEF.md` was generated by `/design-lifecycle`. This documentation and case study were generated by `/project-showcase`. The Validation Report passed all five checks with zero warnings, including a zero-buzzword audit. The self-referential quality is not incidental — it is the clearest signal that the output contracts are coherent.

## Outcomes & Signals

The pipeline is self-validating in a specific sense: the project ran both skills on its own codebase and the outputs passed their respective quality gates. The Validation Report found zero critical failures and zero warnings across accuracy, voice, completeness, specificity, and deck readiness. That includes a line-by-line scan for "leveraged", "robust", "seamless", "best practices" — none found in either artifact.

The Amplifier's tone rules produced a `DESIGN_BRIEF.md` that names specific CSS variable values, specific contrast ratios, specific pixel dimensions — not "comfortable spacing" or "accessible color choices." That specificity is the signal that the anti-AI-voice rules are working as intended.

The six-check Verifier structure in `/design-lifecycle` catches graduated failures rather than producing a binary pass/fail. A fail on WCAG contrast is different from a warn on component coverage, which is different from a pass on internal consistency. The Amplifier can name those distinctions in `DESIGN_BRIEF.md` so the developer knows what requires remediation before implementation versus what is advisory.

This is MVP. The pipeline works well for greenfield projects and applications at early design maturity. For projects with established design systems, the generated token specifications will conflict with existing conventions rather than extend them — that use case requires a different entry point. There is no retry logic, no schema validation on data contracts, and no parallelization. Those limitations are known, named, and documented.

## What I'd Do Differently

The data contract definition was the hardest problem, and I solved it too late. The sequencing mistake was spending time on individual agent prompt quality before locking the interface contracts between agents. Every time I improved the Thinking Partner's output format, I had to update three downstream prompts. Every time I decided the Project Brief needed a new field, the change cascaded. A contract-first approach would have been cleaner: define exactly what the Project Brief contains, what the Inspiration Summary structure looks like, what the orchestrator extracts and re-injects — lock those — then write agent prompts to fill those contracts. The agent prompts become simpler because the output format is pre-defined, not negotiated. I worked from the inside out when the outside should have been the starting point.

I would also invest earlier in making the output format requirements failure-visible. The current enforcement mechanism is asking agents to produce exact heading names in natural language, and trusting that they will. That works until it doesn't, and when it fails, the failure is silent — the orchestrator extracts an empty or malformed block and injects it downstream, and no error appears. A structured output schema, even a simple JSON wrapper around the named fields, would catch format drift at extraction time rather than propagating a broken contract to agents that expect a complete one.

---

## What's Next

- **Structured output contracts.** Replace heading-based extraction with typed JSON blocks in each agent's required output — making the data contract machine-readable and detectable when malformed, rather than pattern-matched from natural language prose.
- **Verifier and Validator as hard blocking gates.** A fail-level finding should stop the pipeline before the Amplifier runs and surface the error explicitly, rather than inheriting it into synthesis. Right now they report; they should also block.
- **Incremental re-runs.** Allow re-running from a specific phase — regenerate only the Maker output after manually adjusting token values, for example — without restarting the entire pipeline from the Thinking Partner.
